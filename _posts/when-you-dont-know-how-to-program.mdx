## プログラミングがわからないときはとりあえずログを出す
なんの言語でもまず、ログを出力しましょう

JavaScriptなら「console.log()」、laravelなら「echo」とか「var_dump」とか。

ログをみないで想像してる場所は想像です。自分は今でも、想像していた結果とログが違うことが毎日、何回もあります。とにかくログに出しましょう。

最初のうちは意味がわからなかった言葉を並べました。こういうものは動かしましょう。自分はこんな感じでやってました。

で、とにかく思いつく限りいろんなパターンを実行します。

もしJavaScriptで配列とか連想配列、オブジェクトがわからない時はこんな感じです。

```javascript
// 配列
const a = [1, 2, 3];

console.log(a[0]);
console.log(a[1]);

// 連想配列
const b = [[0, 1], 2]

console.log(b[0][0]);
console.log(b[1]);

// オブジェクト
const c = {a: 1, b: 2, '2': 3}
console.log(c['a'], c['b'], c.a, c.b, c['2'])
```

laravel

```php
$a = array('a' => 1, 2, 1 => 3);

echo $a['a'];
echo $a[1];
```

async, awaitの理解だけだと限界を感じてPromiseを勉強した時のコードが以下です。githubは[こちら](https://github.com/kajirikajiri/callback-promise-async/blob/master/promise.ts)

```javascript
import { rejects } from 'assert'
import {promisify} from 'util'

const promise1 = (b:boolean) =>{
  return new Promise((resolve,reject)=>{
    setTimeout(()=>{
      if (b) {
        resolve('ok')
      } else {
        reject('ng')
      }
    }, 2000)
  })
}

// https://ja.javascript.info/promise-chaining#ref-220
// thenハンドラは通常、返り値を直ちに次のハンドラにわたします
// しかし、promiseを受け取った場合にはpromiseの処理を待ちます

promise1(true).then(out => {
  console.log(out)
  console.log('then')
  // プロミスを返しているわけではないので次の処理に飛ぶ
  setTimeout(()=>{
    // 四秒後にconsoleが表示される
    console.log(2)
  }, 4000)
}).catch(err => {
  // rejectされないのでcatchに入らない
  console.log(err)
}).then(()=> {
  // プロミスなので２秒後に結果が返る
  return promise1(false)
}).then(ou=>{
  // rejectされるのでthenに入らない
  console.log(ou)
}).catch(err=> {
  // rejectされたのでコンソール
  console.log(err)
})
```

こんな感じで「何がわからないかわかっていれば」とにかく動かして納得できるまでやってみましょう。  
途中で詰まったら、とりあえずネットから動くものをコピペしてきてそれを元に色々動かしましょう。これできっと大丈夫!

## プログラミングのなにがわからないのかわからないとき

始めたばかりだとよくあります。自分も最初の頃はどのプログラムが原因なのかわからないけど、よく粘ってました。１週間〜長くて１ヶ月くらいは一つの問題で悩んで結局「わかんないな」ってなって諦めたりもしました。この前Twitterで１週間くらい詰まっている方がいたのでちょっとみてきたのですが、最初のうちは難しいだろうなっていう問題でした[^1]。方法は色々あります。

1. 諦める
1. 頼る
1. 粘る

### 諦める

こんな感じだと思います。とりあえず「諦める」には２種類あります。

1. 完全に諦める
1. この手法は諦めて別の選択肢を探る

の２種類です。まずは別の手法を探してみましょう。しかし、最初のうちはこれが難しいです。自分の場合ですが、１つのサイトを参考にずっと悩んでいました。これはやめましょう。多分サイトの内容が古くなっているか、書いた人の記載漏れだと思って別のものを探しましょう。ググっているなら1番目の結果だけでなく、２番目３番目、５番目くらいまでみてみましょう。  
これでダメな時はググり方が悪い可能性もあります。検索する言葉を変えましょう。日本語使ったり英語使ったり、検索ワードの順番を変えたり、検索する期間を変更したり、double quoteで完全一致検索しましょう。大体は見つかります。  

google 検索は[こちらのhubspot](https://blog.hubspot.jp/update-how-to-search-in-google-x-advanced-google-search-tips-0-0)の記事が参考になりそうです

あと最初の頃は見逃していたのですが、Googleで検索した結果に出てくるStackOverflowの検索結果は、一つの下に５個くらい類似の記事が紐づいています。これらは割と正解なので、開いてみましょう。  

英語わかんないってなったら[DeepL](https://www.deepl.com/translator)を使いましょう。

完全に諦めるについてですが、業務じゃなければとりあえずありです。業務だと、これが必要なんだ！っていう場合があり、そう言う場合は逃げられません、そこでググり力というか検索力が上がります。業務じゃなければ、まあいつかその問題にぶつかるので、その時にがんばりましょう！今はできるだけやったんです!  

### 頼る

「頼る」ですが、「[MENTA](https://menta.work/)」とか色々あると思うので使ってみましょう。知っている人なら本当に一瞬で解決してくれると思います。自分は心理的ハードルが高かったのと存在をあまり知らなかったので使いませんでしたがお勧めします。ある程度プログラマ経験があるひとならその言語を知らなくても、話してるうちに解決してくれると思います。書いてて思いましたが、自分も新しいものを学ぶときは頼った方が早いかなって思えてきました。心理的ハードルが高い方は自分に[twitterでDM](https://twitter.com/kajirikajiri)いただいてもokです。質問して貰えば私の経験値にもなるんでむしろお願いします。

### 粘る

「粘る」ですが、上記で書いたように業務だと逃げられない可能性が高いです。そういう状況を想定して粘ってみるのもありです。ただ、自分の例で紹介したような、１つの問題に１ヶ月とかかけるのは根気がいると思うので、どこかで諦めるのもありだと思います。

[^1]: Mysqlの接続先を指定する箇所を修正すれば良いという問題でした。サーバーサイド・バックエンドあたりの知識が必要な問題だったので、「問題箇所と問題内容の特定」・「対処法」がわからなければいけないです。laravelを使用したシステムでしたが、railsとかnodeとかでサーバーサイドを一度経験済みでないと難しいだろうなと思いました。gcpとかheroku、awsで無料でサーバーを起動して、フロントをローカルホストから接続。成功したらフロントもデプロイして接続（ここでcorsとかでつまるはず）してみて、成功したら消してを繰り返せばそのうち覚えられます。自分はそうやりました。が、自分はフロント側をメインでやっていたので辛かったです。特にfirewallに苦しめられました。

## ずっと同じエラーで動かなくてプログラミングがわからないとき

先ほど例としてあげましたが、自分も長くて1ヶ月くらいプログラムが動かなくて詰まったことがあります。特に環境構築ですね。vuetifyとか、dockerとか、この前だとelectronとかflutterでのアプリ開発環境構築です。このような環境構築しているときに、ずっと同じエラーで詰まっていることがよくありました。この時の対処法として２つあります。

- 別のエラーを発生させる
- 本当の原因を探る
- エラーメッセージをよくみる

### 別のエラーを発生させる

まず、「別のエラーを発生させる」ですが、同じエラーが発生し続けて、しかもググってもわからないという場合、まずエラーが変わるようにしましょう。その部分にずっと対応していても解決しない可能性が高いです。もし、もともと発生していなかったエラーだとすれば、その時点まで戻って確認してみましょう。何か不要なコードがあるかもしれません。  

### 本当の原因を探る

「本当の原因を探る」ですが、今あなたはここが原因だ！と思っている箇所があると思います。そこを一旦全部削除してみましょう。そして動かしましょう。これで同じエラーが出る場合、原因は他の場所にある可能性が高いです。そのあと、原因になりそうな部分を半分消して動かして、半分消して動かしてを繰り返してみてもいいと思います。これでエラーメッセージが変わればそこらへんに原因があるかもしれません。

### エラーメッセージをよくみる

「エラーメッセージをよくみる」ですが、エラーメッセージにはよくERRORとかWARNINGが先頭についています。ERRORは赤色、WARNINGは黄色で表示されていることが多く、ERRORの方が重要に見えます。しかし、WARNINGもみてみましょう。実はWARNINGに本当の原因が書いてあることもあります。自分もrubocopやelectonがわからない時にWARNINGをみたら解決したことがあります。

## フレームワークが動かなくてプログラミングがわからないとき

フレームワークが動かない時も辛いです。この前は[parallax.js](https://github.com/wagerfield/parallax)を動かせなかったです。こういう時の対処としていくつかあります。

- コピペをちゃんとしたか確認する
- 発生したエラーでググる
- ソースコードを見る

### コピペをちゃんとしたか確認する

「コピペをちゃんとしたか確認する」ですが、色々試しているうちにプログラムが書き変わってしまっていて、動かなくなっていることがあるので目視や差分確認ツールで確認することがいいと思います。  

### 発生したエラーでググる

また、「発生したエラーでググる」ですが、例えばCATというフレームワーク(例です)が動かないときに「CAT エラー」とか「CAT 動かない」でググっても答えは出てきません。黒い画面（コンソールとかターミナルとかいうやつ）に出てくる長いエラーメッセージをそのまま貼り付けましょう。  
例えば自分の経験としてはtigというツールを使っていてこんなエラーが出ました

```
tig: No revisions match the given arguments. (unknown revision or path not in the working tree.) 
```

これをこのまま貼り付けましょう。それで大体は解決します。  

時々何も見つからないのですが、その時はエラーメッセージから不要な部分を除去します。ファイル名が不要なことが多いです。例えば以下のエラーがあったとします。  

```
Unknown file name "/Users/kajiri/ghq/github.com/kajiri/github/vscode-plugins-2021.js"
```

この場合、"/Users/kajiri/ghq/github.com/kajiri/github/vscode-plugins-2021.js" が不要なことが多いです。理由として、

- そのファイルはあなたの環境にしか存在しない
- double quote(")はGoogle検索で特別な意味をもつ

ことが理由にあります。ファイル名の部分は削除して、double quote(")も削除しましょう。これでいい結果が出てくる時があります。

google 検索は[こちらのhubspot](https://blog.hubspot.jp/update-how-to-search-in-google-x-advanced-google-search-tips-0-0)の記事が参考になりそうです

### ソースコードを見る

また、これはちょっと難しいのですが、「ソースコードを見る」ことで解決できる可能性もあります。自分もこの方法を最初試した時は挫折しました。無理でした！。１行進むごとに知らない構文が出てきて、さっきどこまで読んだっけ？みたいになるからです。ブックマークしておいて時々読むとか、MENTAとかでわかる人に聞きましょう。  

一応ソースコードをみる方法はいくつかあります。

- github上でフレームワークのソースコードをそのまま見る
- フレームワークをgithubからcloneしてローカルで確認する
- フレームワークをgithubからforkしてローカルで動かす

プログラムが読めるようになってくればソースコード上から判断することも可能です。
